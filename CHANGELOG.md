# Журнал изменений RiftLink

Формат: версия **прошивки и приложения** синхронизированы по семантике релиза (например `1.5.19`).

## 1.5.19 — 2026-03-23

### Надёжность личных сообщений и BLE

- **Входящие сообщения (`evt:msg`)** — исправлена дедупликация в приложении: при отсутствии стабильного `msgId` в JSON разные сообщения с одинаковым текстом больше не схлопываются; при валидном `msgId` повторная доставка по-прежнему отфильтровывается. Ограничен рост кэша ключей дедупа при длительной сессии.
- **`getInfo`** — при троттлинге запроса повторно выдаётся последний снимок `info` в шину событий, чтобы экраны (соседи, настройки) не «застывали», если свежий JSON уже есть в памяти; при **полностью недоступном транспорте** (нет BLE/Wi‑Fi) в шину снова отдаётся последний закэшированный `evt:info`.
- **Экран чата** — при возврате приложения с фона список сообщений перечитывается из локальной БД, чтобы устранить расхождение «в базе есть — в открытом чате нет»; то же при **возврате с вложенного экрана** поверх чата (например диалог), через глобальный `RouteObserver`.
- **Отправка в личный чат** — при невалидном ID получателя (не 16 hex) в лог добавляется явная причина `invalid_recipient` вместо немого `send() == false`.
- **Ping / проверка «онлайн»** — исправлена логика `sendPing`/`sendPingTracked`: раньше признак успеха почти всегда был «да», даже если BLE не принял JSON (ошибка глоталась); теперь коротко ждём ответ роутера и отличаем `request_send_failed` от ожидания pong по эфиру. После подключения `getInfo` → `getGroups` → `getRoutes` выполняются **по очереди**, чтобы реже переполнять очередь команд на узле. Авто-ping при входе в личку задержан (~520 ms). В списке чатов: таймаут ожидания pong увеличен до 20 s, сопоставление ответа с ожидаемым ID — с учётом префикса (как в экране чата).
- **Traceroute / selftest** — увеличены таймауты ожидания ответа по BLE (`traceroute` 18 s, `selftest` 15 s).
- **Прошивка: `signalTest`** — увеличены задержки между отложенными `OP_PING` по соседям (реже коллизии в эфире и лавина `notify` на BLE).
- **Экран Mesh / Signal Test** — перед запуском теста запрашивается свежий `getInfo(force: true)`; при пустом списке соседей — повтор через **1 с** и подсказка «на узле нет соседей»; сопоставление `from` в `evt:pong` с соседями по **очищенному hex** (без лишних символов); окно сбора ответов **10 с**; ошибка прошивки `signal_test_no_neighbors` показывает ту же подсказку.
- **Личный чат** — убран второй авто-ping при каждом `evt:info` (оставлены вход в чат / таймер 15 s / возврат с фона); подпись «проверяем связь» без подстановки Node ID.
- **Диагностика** — дополнительная строка лога для цепочки прошивка → ingest → репозиторий по входящим сообщениям.
- **BLE: разбор крупного `evt:info`** — при склейке нескольких notify промежуточный `FormatException` (обрыв строки в середине JSON) больше не засоряет `[BLE_CHAIN] app_parse single_object_retry`; полный список групп по-прежнему доступен через `getGroups`.
- **Прошивка: `notifyInfo` по BLE** — в JSON ограничено число элементов `groupsV2` (до 12), чтобы реже резать полезную нагрузку на границе MTU; пауза между chunk’ами notify увеличена с 8 до 12 ms.
- **Прошивка: `evt:pong` по BLE** — при приёме `OP_PONG` **сразу** `notifyJsonToApp`, как для `evt:delivered` после `OP_ACK` (раньше pong всегда ждал следующий `ble::update()` и из‑за этого ощущался медленнее доставки сообщений). Если мьютекс notify занят — отложенная отправка через `update()` (как раньше), один notify за тик, приоритет после `info` и **раньше тяжёлого `evt:msg`**; `cmdId` не снимается до успешной отправки (`peek` / `take` после OK).
- **Прошивка: ретраи `OP_PING` по эфиру** — меньше дублей, чем у MSG: **до трёх передач** одного запроса (первая + 2 повтора); таймаут как у ACK в очереди сообщений; таймаут считается по **max(SF модема, подсказка RSSI→SF)** — фактический TX с **фиксированным SF модема**; при приёме **`OP_PONG`** от того же узла ретраи сбрасываются (`clearPingRetryForPeer`).
- **Прошивка: HELLO / beacon** — реже лишний трафик в эфире: базовый интервал 36 с (было 30 с), при 0 соседей агрессивный режим 12 с (было 9 с), ступени после 2/5 мин без соседей 15/18 с (было 12/15 с), при одном соседе 24 с (было 20 с), при ≥6 соседей 30 с (было 24 с), жёсткий минимум между HELLO 10 с (было 8 с).
- **Приложение: `TransportResponseRouter` и `evt:pong`** — если в JSON нет `cmdId` (прошивка не вложила), ответ всё равно сопоставляется с ожидающим tracked `ping` по полю `from` и аргументу `to` команды (полный id / префикс), чтобы тикет не «висел» вечно.
- **Прошивка: JSON `evt:pong` по BLE** — буфер сериализации увеличен до 128 байт, вызов `serializeJson` с границей; при переполнении notify не отправляется (исключена обрезка JSON и «пустой» ping).
- **Прошивка / приложение: связка пинг ↔ pong** — в ответном кадре `OP_PONG` в эфире передаётся тот же `pktId`, что был в `OP_PING` (если исходный пинг с pktId); в `evt:pong` добавлено поле **`pingPktId`** (дополнительно к `cmdId` из BLE-команды). Во Flutter — `RiftLinkPongEvent.pingPktId`.

### Тесты и инфраструктура

- Тесты на ключ дедупа входящих сообщений; исправлены ожидания в тестах навигации (дублирование заголовка на тестовой странице).
- **TransportResponseRouter** — сценарий `pong` без `cmdId` для tracked-команды `ping`.

### Веб-прошивка (ESP Web Flasher)

- Обновлены манифесты и образы `*_full.bin` для Heltec V3, V3 Paper и V4.
