# Смоук: цепочка приложение ↔ узел

Автоматическая статическая проверка: `python scripts/verify_app_node_chain.py` (из корня репозитория).

Ниже — ручные шаги на железе; без устройства их не выполнить.

## BLE (минимум)

1. Включить узел (ESP, прошивка RiftLink), открыть приложение, найти `RL-…`, подключиться.
2. Убедиться, что после подключения приходят события (лог отладки / экран статуса): не пустой поток JSON.
3. Дождаться волны `info` (или потянуть обновление узла): должны быть `evt: node` и (при tracked) `cmdId` в нескольких событиях подряд (`node`, `neighbors`, `routes`, `groups`).
4. Отправить broadcast-сообщение: ожидается `evt: sent` или осмысленный `evt: error`.
5. При втором узле в сети: входящее `evt: msg`.

## Wi‑Fi (опционально, тот же JSON)

1. Переключить узел в режим STA (команда `radioMode` / настройки Wi‑Fi в приложении — как в UI).
2. Подключиться к WebSocket узла (как в `docs/API.md` / код приложения для Wi‑Fi режима).
3. Повторить пункты 3–4 из секции BLE (снимок `info`, отправка сообщения).

## nRF52840 (FakeTech / T114)

Паритет BLE: см. `docs/API.md` §3.16 (таблица `ota_unsupported`, `powersave_unsupported`, …). Команды `bleOta*` на nRF не поддерживаются; на ESP — поддерживаются. Статический diff `strcmp(cmd)`: вывод скрипта `verify_app_node_chain.py` (`only_ESP` / `only_nRF`).
