### 📌 Текущее состояние (прошивка 1.3.5)

Основной функционал реализован: MSG, X25519 E2E, GROUP_MSG, офлайн-очередь, ROUTE_REQ/REPLY, VOICE_MSG, ACK/READ, LZ4, MSG_FRAG, LOCATION, TELEMETRY, OTA, регионы, invite/acceptInvite (QR). Валидация пакетов при RF-помехах. Поддержка V3 (OLED), V4 (OLED), V3 Paper (E-Ink).

- [x] **Сжатие** — LZ4 для текста ≥50 байт
- [x] **Фрагментация** — разбиение сообщений >200 байт на части, сборка на приёмнике
- [x] **Голосовые сообщения** — запись → Opus 8 kbps → отправка → воспроизведение

> **Важно:** Все сообщения (текст, голос, фрагменты) передаются только в зашифрованном виде (E2E). Порядок: **Сжатие → Шифрование → Фрагментация** (сжатие до шифрования — зашифрованные данные не сжимаются).

## 4. 🏗️ Архитектура

### 4.1 Стек протокола

```
┌─────────────────────────────────────────────────────────┐
│  Приложение (Android/iOS/Desktop)                        │
│  - Чат, карта, настройки                                 │
└───────────────────────┬─────────────────────────────────┘
                        │ BLE / Serial (JSON или бинар)
┌───────────────────────▼─────────────────────────────────┐
│  Прошивка (ESP32)                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ │
│  │ App Layer  │  │ Crypto      │  │ Routing         │ │
│  │ (сообщения)│  │ (E2E)       │  │ (mesh)          │ │
│  └──────┬─────┘  └──────┬──────┘  └────────┬────────┘ │
│         │               │                   │          │
│  ┌──────▼───────────────▼───────────────────▼────────┐  │
│  │ Packet Layer — формат пакета, opcodes             │  │
│  └──────┬────────────────────────────────────────────┘  │
│         │                                                │
│  ┌──────▼────────────────────────────────────────────┐  │
│  │ Radio Layer — LoRa (SX1262), частоты, SF, BW       │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 4.2 Формат пакета

**v2 (version 0x20, с 1.2.6):** opcode первым, компактный broadcast (экономия 8 байт).

**Broadcast** (HELLO, PING, ROUTE_REQ и т.д., флаг broadcast=1):
```
┌──────┬────────┬────────┬────────┬──────┬────────┬────────┐
│ Sync │ Ver    │ Opcode │ From   │ TTL  │ Channel│Payload │
│ 1 B  │ 1 B    │ 1 B    │ 8 B    │ 1 B  │ 1 B    │ N B    │
└──────┴────────┴────────┴────────┴──────┴────────┴────────┘
```
Заголовок: **13 байт** (без 8 байт to=0xFF..FF).

**Unicast** (MSG, ACK, KEY_EXCHANGE и т.д.):
```
┌──────┬────────┬────────┬────────┬────────┬──────┬────────┬────────┐
│ Sync │ Ver    │ Opcode │ From   │ To     │ TTL  │ Channel│Payload │
│ 1 B  │ 1 B    │ 1 B    │ 8 B    │ 8 B   │ 1 B  │ 1 B    │ N B    │
└──────┴────────┴────────┴────────┴────────┴──────┴────────┴────────┘
```
Заголовок: **21 байт**.

Ver: версия (4 bit) + флаги (encrypted, compressed, ack_req, broadcast)
From/To: Node ID (8 байт, первые 4 = short ID для роутинга)
Opcodes: 0x01 MSG, 0x02 ACK, 0x03 HELLO, 0x04 ROUTE_REQ, 0x05 ROUTE_REPLY, 0x06 KEY_EXCHANGE, ...

### 4.2.1 Максимальное число хопов

**Да, делаем максимум.** TTL = 1 байт → до **255 хопов** теоретически.

| Параметр | Значение |
|----------|----------|
| **Макс. TTL по умолчанию** | 255 (или 31 для экономии — 5 бит) |
| **Настройка** | `max_hops` в конфиге: 5, 15, 31, 255 |
| **При ретрансляции** | Каждый узел уменьшает TTL. При TTL=0 пакет отбрасывается |

**Практика:** При 255 хопах один пакет может идти минуты (LoRa медленный). Рекомендуется настраиваемый лимит: 15–31 для обычной работы, 255 — для «максимальной дальности» (эксперименты, редкие сообщения).

### 4.3 Opcodes (расширяемый список)

| Opcode | Название | Описание |
|--------|----------|----------|
| 0x01 | MSG | Текстовое сообщение (шифр.) |
| 0x02 | ACK | Подтверждение доставки |
| 0x03 | HELLO | Анонс узла (beacon) |
| 0x04 | ROUTE_REQ | Запрос маршрута |
| 0x05 | ROUTE_REPLY | Ответ с маршрутом |
| 0x06 | KEY_EXCHANGE | Обмен ключами (X25519) |
| 0x07 | TELEMETRY | Телеметрия (шифр.) |
| 0x08 | LOCATION | Геолокация (шифр.) |
| 0x09 | GROUP_MSG | Групповое сообщение (шифр.) |
| 0x0A | MSG_FRAG | Фрагмент длинного сообщения (шифр.) |
| 0x0B | VOICE_MSG | Голосовое сообщение (шифр.) |
| 0x0C | READ | Подтверждение прочтения (payload: msg_id 4B) |
| 0x0D | NACK | Запрос повтора (payload: pktId 2B, v2.1) |
| 0x0E–0xFD | Reserved | Для будущего |
| 0xFE | PONG | Ответ на PING |
| 0xFF | PING | Служебный ping |

### 4.4 Сжатие сообщений

**Да, сжатие возможно и полезно** — особенно для текста и JSON. Снижает число пакетов и время передачи.

**Порядок операций (обязательный):**
```
[Исходные данные] → [Сжатие] → [Шифрование] → [Фрагментация] → LoRa
```
> Зашифрованные данные не сжимаются (выглядят как случайные). Сжимать нужно до шифрования.

| Алгоритм | Плюсы | Минусы | Рекомендация |
|----------|-------|--------|--------------|
| **LZ4** | Быстрый, мало RAM, ~2x для текста | Меньше сжатие чем zlib | ✅ Для ESP32 |
| **miniz/deflate** | Хорошее сжатие (~3x для текста) | Больше CPU и RAM | Альтернатива |
| **LZ4HC** | Лучше сжатие | Медленнее | Для длинных сообщений |

**Когда применять:**
- **Текст** — всегда (сокращение 50–70%)
- **JSON** — да (много повторяющихся ключей)
- **Голос (Opus)** — нет (уже сжат кодеком)
- **Короткие сообщения (<50 байт)** — опционально (overhead может увеличить размер)

**Флаг в Header:** `compressed=1` — получатель знает, что payload нужно распаковать после расшифровки.

---

### 4.5 Фрагментация длинных сообщений

**Ограничение LoRa:** ~255 байт на пакет, полезная нагрузка ~200–250 байт.

**Схема:** Сообщение сжимается (если нужно) → шифруется → разбивается на фрагменты.

```
[Исходное сообщение] → [Сжатие] → [E2E шифрование] → [Фрагмент 1][Фрагмент 2]...
       ↓                    ↓              ↓                    ↓
   Текст/данные         LZ4 (опц.)   ChaCha20-Poly1305      Пакеты MSG_FRAG
```

**Формат MSG_FRAG (payload после расшифровки):**
```
┌────────────┬────────────┬────────────┬─────────────┐
│ MsgID 4B   │ Part 1B    │ Total 1B   │ Data N bytes│
└────────────┴────────────┴────────────┴─────────────┘
```
- **MsgID** — идентификатор сообщения (связывает фрагменты)
- **Part** — номер фрагмента (1..Total)
- **Total** — всего фрагментов
- **Data** — часть зашифрованного сообщения

**Важно:** Шифруется всё сообщение до разбиения. Роутеры видят только зашифрованные фрагменты.

---

### 4.6 Голосовые сообщения

**Режим:** Асинхронный (запись → отправка → воспроизведение), как в мессенджерах.

**Цепочка:**
```
[Запись на телефоне] → [Opus 8 kbps] → [E2E шифрование] → [Фрагментация] → LoRa
                                                                              ↓
[Воспроизведение] ← [Декодирование] ← [Сборка + расшифровка] ← [Приём]
```

| Параметр | Значение |
|----------|----------|
| Кодек | Opus 8 kbps |
| 10 сек | ~10 KB ≈ 40–50 пакетов |
| 30 сек | ~30 KB ≈ 120–150 пакетов (лимит) |
| Шифрование | Всё тело сообщения (аудио) — E2E |

**Opcode VOICE_MSG:** Метаданные (длительность, кодек) в заголовке; аудиоданные — зашифрованный payload, фрагментированный через MSG_FRAG.

### 7.1 Flutter и PWA

**Flutter** выбран для мультиплатформенности:
- **Android / iOS** — нативные приложения через BLE
- **Web (PWA)** — работает в браузере, можно установить на домашний экран; Web Bluetooth API для связи с устройством (Chrome, Edge)
- **Desktop** — Windows, macOS, Linux (опционально, для отладки и карты сети)

**PWA (Progressive Web App):**
- Flutter поддерживает сборку в PWA (`flutter build web`)
- Service Worker, офлайн-кэш, установка без магазина
- Web Bluetooth ограничен: Chrome/Edge на Android, частично на desktop; Safari — ограниченная поддержка
- **Рекомендация:** PWA — для быстрого доступа и демо; нативные Android/iOS — основной сценарий (Web Bluetooth стабильнее)

## 11. 💡 Идеи для расширения (фаза 3+)

### Шлюзы и интеграция

| Технология | Описание | Сложность |
|------------|----------|-----------|
| **MQTT** | Pub/Sub через mesh или шлюз. Топики `node/123/msg`, `node/123/telemetry`. Интеграция с Home Assistant, Node-RED | Средняя |
| **Wi‑Fi Gateway** | Heltec V4 имеет WiFi. Узел с интернетом — шлюз mesh↔интернет. Отправка в облако, Telegram-бот | Средняя |
| **HTTP API** | Web-интерфейс при подключении по BLE/WiFi. Настройки, история, карта узлов | Низкая |
| **WebSocket** | Реалтайм при наличии WiFi. Стриминг статусов, уведомлений | Низкая |

### Локальная связь

| Технология | Описание | Сложность |
|------------|----------|-----------|
| **ESP-NOW** | Встроенный в ESP32. Быстрая связь до ~200 м (2.4 GHz). Дублирование критичных сообщений LoRa+ESP-NOW | Низкая |
| **BLE Mesh** | Устройства рядом — обмен через BLE без LoRa. Экономия энергии, быстрее | Высокая |
| **NFC** | Tap-to-pair: обмен ключами касанием телефонов. Безопасная инициализация | Низкая |
| **QR-коды** | Показ/сканирование Node ID, публичный ключ, приглашение в канал | Низкая |

### Прошивка и обновления

| Технология | Описание | Сложность |
|------------|----------|-----------|
| **OTA по mesh** | Обновление прошивки через LoRa: передача образа по частям, проверка CRC | Высокая |
| **DFOTA** | Differential OTA — только дельта изменений. Меньше трафика | Высокая |
| **A/B разделы** | Безопасное обновление: новая прошивка в backup, переключение при успехе | Средняя |

### IoT и данные

| Технология | Описание | Сложность |
|------------|----------|-----------|
| **Cayenne LPP** | Стандартный формат телеметрии (LoRaWAN-совместимый). Датчики, батарея, GPS | Низкая |
| **Store-and-Forward** | Узел хранит сообщения для офлайн-узлов. Доставка при появлении | Средняя |
| **Time-sync** | Синхронизация времени по mesh (NTP-подобный). Метки времени для сообщений | Низкая |
| **Sigfox/LoRaWAN** | Мост в коммерческие сети (если нужна интеграция) | Высокая |

### Безопасность и анонимность

| Технология | Описание | Сложность |
|------------|----------|-----------|
| **Forward secrecy** | Смена ключей по сессиям. Компрометация одного ключа не раскрывает историю | Средняя |
| **Псевдонимы** | Временные ID для анонимности. Ротация идентификаторов | Средняя |
| **Plausible deniability** | Скрытые сообщения (steganography). Сложно доказать наличие | Высокая |

### Приложение

| Технология | Описание | Сложность |
|------------|----------|-----------|
| **PWA** | Flutter Web → PWA. Работает в браузере, установка без магазина. Web Bluetooth (Chrome) | Низкая |
| **Desktop** | Electron/Tauri для ПК. Полноценный клиент с картой | Средняя |
| **Wear OS** | Уведомления на часы. Короткие сообщения с экрана | Средняя |

### Приоритеты для «крутых» фич

1. **MQTT** — интеграция с умным домом, дашборды
2. **Wi‑Fi Gateway** — выход в интернет, Telegram-бот
3. **ESP-NOW** — гибрид LoRa+ESP-NOW для надёжности
4. **NFC/QR** — удобный pairing
5. **OTA по mesh** — обновления без проводов
