Формат: версия **прошивки и приложения** синхронизированы по семантике релиза (например `1.5.20`).

## 1.5.19 — 2026-03-23

### Приложение: список групп не затирается пустым `evt:groups`

- Склейка `evt:node` + … + `evt:groups` иногда давала **второй** снимок `groups: []` (гонка с `PEND_INFO` / `getGroups` / кратковременное состояние на узле). В `_mergeFromGroups` пустой массив **полностью заменял** уже непустой список — группы **мигали** и **пропадали**. Если приходит **пустой** снимок, а в композите уже есть группы — **сохраняем** предыдущий список (полный сброс групп на устройстве до следующего непустого снимка или переподключения).
- После **`groupCreate`** узел шлёт **`groupStatus`** раньше, чем в композит попадает полный **`evt:groups`**. Следующий `evt:node` без поля `groups` давал **пустой** список в склеенном `RiftLinkInfoEvent` и **затирал** экран групп, хотя `lastInfo` уже должен был знать группу. **`evt:groupStatus` теперь вливается в композит и `lastInfo`** (как минимальный снимок группы), затем `evt:node` сохраняет непустой список. Парсинг `groupTag` в `groupStatus` для склейки.
- Сообщения **`group_v3_store_failed` / `group_v2_store_failed`** (ошибка NVS / «store») вынесены в **локализованные** строки вместо сырого `Failed to store…`.

### Группы: свой инвайт «сам себе» не перезаписывает роль

- **Прошивка** — после проверки подписи `groupInviteAccept`: если `groupUid` **уже** есть в NVS, **не** вызывается `upsertGroupV2` (роль и ключ из инвайта не подменяются). Отправляется `evt: groupStatus` с `inviteNoop: true`, обновление списка групп как при успехе.
- **Приложение** — строка «Группа уже есть на этом устройстве — роль и инвайт не менялись»; флаг `takeGroupInviteAcceptWasNoop()` после успешного приёма.

### BLE / группы: один массив `groups`

- Удалён legacy-массив transport id; в JSON и в приложении поле **`groups`** — только список объектов (`groupUid`, `channelId32`, …). Бывший `groupsV2` в протоколе не используется; класс в Dart переименован в **`RiftLinkGroupInfo`**.
- **Инвайт в группу** — копирование и связанные действия на экране групп / в списке чатов проверяли только `isConnected` (BLE GATT). В режиме **Wi‑Fi/WebSocket** транспорт активен, а BLE может быть «отключён», из‑за чего «Скопировать инвайт» молча не выполнялось. Используется **`isTransportConnected`**; при отсутствии транспорта показывается подсказка «Сначала подключитесь к устройству».
- **Статус ключа после вступления** — в прошивке `rekeyRequired` = `!ackApplied`; после приёма инвайта ключ новый и **ACK ещё не отправлен**, поэтому флаг не означает «владелец должен сделать rekey». Строки в UI переименованы: **«Подтвердите ключ»** вместо вводящего в заблуждение «Требуется rekey».
- **Создание группы** — пользователь вводит только **название**; числовой **`channelId32`** (2…0xFFFFFFFF) выбирается **случайно** с проверкой на пересечение с уже известными на устройстве группами (экран групп и FAB в списке чатов). Символ `|` в названии на стороне приложения убирается (ограничение прошивки).
- **«Скопировать инвайт» / усечённый код** — ответ `evt: groupInvite` сериализовался в буфер **720 байт**; при длинном BASE64 и имени группы JSON **обрезался**, в буфер обмена попадала **неполная** строка (не ошибка копирования во Flutter). Буфер увеличен до **2048**, используется `serializeJson(..., sizeof(out))`.
- **Инвайт в группу по вставке** — при вставке из мессенджера в BASE64 часто попадают **переносы строк и пробелы**; прошивка декодировала как «Bad invite base64», в UI показывалось «Некорректный payload инвайта группы». Добавлена **нормализация** строки (удаление пробелов/переводов, url-safe BASE64, паддинг `=`) перед `groupInviteAccept`; уточнены сообщения об ошибках и таймаут без дублирования с `groupSecurityError`.
- **Создание группы / «отсутствует ID»** — для больших `channelId32` JSON приходит как число вне диапазона `int32`; прошивка брала `channelId32` через `| 0` и получала **0**, ответ `groupSecurityError` с текстом про Missing… **и** таймаут tracked-команды в приложении. Исправлено: на прошивке разбор `channelId32` через `double`/строку; в приложении `TransportResponseRouter` завершает ожидание по `evt: groupSecurityError`, русские подсказки вместо сырого `Missing groupUid/…`.

### BLE: раздельные события вместо монолитного `evt:info`

- **Прошивка** — ответ на `cmd: info` разбит на волну с общим `seq`: `evt:node` (паспорт без тяжёлых массивов), затем `neighbors` / `routes` / `groups` с тем же `seq`. Убран буфер «последнего гигантского JSON» как единая точка отказа.
- **Приложение** — частичные evt склеиваются в один `RiftLinkInfoEvent`; tracked-команды, ожидавшие `evt:info`, переведены на первый кадр волны (`evt:node`). Экраны Mesh/чаты/контакты/группы слушают обновлённый снимок через `RiftLinkInfoEvent` (отдельные `Routes`/`Groups`/`Neighbors` в шине после склейки не дублируются).

### Прошивка: группы и инвайт

- **`groupInviteAccept`** — после успешной записи ключа из проверенного инвайта узел сразу вызывает **`ackKeyAppliedV2`** для текущей версии ключа. Раньше `upsertGroupV2` оставлял `ackApplied=0` до отдельной команды с телефона; для вступления по инвайту отдельный шаг не нужен (ключ уже применён в этом же сеансе). Ручной ACK по-прежнему актуален после **смены ключа (rekey)**, когда участник получает новую версию не через инвайт.
- **`groupCreate`** — то же: после локальной генерации ключа вызывается **`ackKeyAppliedV2`**, чтобы в `groupStatus` / списке групп сразу был **ключ актуален** (`rekeyRequired: false`), без лишнего нажатия «Подтвердите ключ» на телефоне.

### Прошивка: обмен ключами и очередь TX

- **KEY_EXCHANGE / отложенный TX** — при переполнении слотов `queueDeferredSend` резервный путь ошибочно помечал пакет как `data`, а не `control`: ломался профиль асимметрии (`computeTxAsymSkewMs`) и поведение приоритетной очереди; handshake мог «застревать», соседи выпадали по таймауту. Исправлено: класс берётся из opcode, как у обычного deferred; то же для fallback у `queueDeferredRelay`. Увеличено число слотов отложенной отправки (24), чтобы реже уходить в fallback при всплесках.

### Прошивка: BLE — доставка и прочтение в приложении

- **`evt:delivered` / `read` / `sent` / `undelivered` / broadcast delivery** — при занятости notify mutex длинным `evt:info` (несколько chunk’ов) мелкие события раньше **терялись** без повтора; в UI это выглядело как «галочки не приходят» или приходят с огромной задержкой. Добавлено кольцевое буферирование (8 слотов) и повторная отправка в начале `ble::update()` до успешного notify, по аналогии с отложенным `pong`. Кольцо защищено отдельным mutex (producer/consumer в разных задачах); это **не** относится к lock-free очереди `send_overflow` для эфирного TX.

### Прошивка: BLE — скорость notify на телефон

- **Между chunk длинного JSON** убрана фиксированная пауза **2 ms** (при ~3 chunk для `info` это ~4 ms только из sleep). По умолчанию: **`taskYIELD()`** без миллисекунд; при необходимости задать паузу — `-DRIFTLINK_BLE_NOTIFY_INTER_CHUNK_DELAY_MS=1`. Подробный лог каждого notify в Serial по умолчанию **выключен** (`RIFTLINK_BLE_CHAIN_NOTIFY_LOG=1` — для отладки).

### Приложение: ложные «отвалы» BLE

- **Watchdog транспорта и чата** — исправлено: при успешном `getInfo` флаг probe не сбрасывался; через ~5 с срабатывал `probe_timeout` и шло переподключение при живом GATT. Ошибка/таймаут `getInfo` при **ещё подключённом** BLE больше не трактуется как разрыв (ответ мог не успеть из‑за длинного notify). Порог тишины **25 s**, ожидание завершения probe **12 s**.
- **Двойной idle-probe** — при активном `TransportReconnectManager` экран чата больше не запускает свой же idle/`getInfo` по тому же таймеру (~1.2 с), чтобы два параллельных probe не перегружали BLE и не усугубляли таймауты.
- **Смена узла с экрана сканирования** — намеренное отключение от предыдущего устройства при подключении к другому больше не запускает ложный авто‑reconnect к старому узлу: подавление reconnect на время `RiftLinkBle.connect()` и снятие с задержкой ~450 ms; внутренние попытки менеджера reconnect и оверлея вызывают `connect(..., internalReconnect: true)`, чтобы не сбрасывать FSM; убран ранний `resumeAutoReconnect()` в начале подключения на скрине.

### Приложение: покинуть группу из списка чатов

- **Unknown group** при `groupLeave` — для чатов, созданных из сообщений **без поля `groupUid`** в `evt:msg`, в БД сохранялся плейсхолдер `uid:UNRESOLVED_<channelId>`. В узел уходил не настоящий UID, прошивка отвечала ошибкой. Разрешение UID через `lastInfo.groups` / запрос `getGroups`, отказ отправлять строки `UNRESOLVED_*` в `groupLeave`, подсказка при невозможности определить UID.

### Надёжность личных сообщений и BLE

- **`evt:neighbors`** — пустой список сразу после `evt:info` с непустыми `neighbors` затирал соседей в UI (чат, Mesh, контакты); при таком рассинхроне подставляется снимок из `lastInfo` и запрашивается сверка `getInfo(force: true)`. Дополнительно: если локальный список пуст, а в `RiftLinkBle.lastInfo` уже есть соседи — подтягивание из кэша (таймер чата, возврат с фона, Mesh после `getInfo`). Разбор поля `neighbors` в JSON допускает не только массив (строка / список через запятую).
- **Входящие сообщения (`evt:msg`)** — исправлена дедупликация в приложении: при отсутствии стабильного `msgId` в JSON разные сообщения с одинаковым текстом больше не схлопываются; при валидном `msgId` повторная доставка по-прежнему отфильтровывается. Ограничен рост кэша ключей дедупа при длительной сессии.
- **`getInfo`** — при троттлинге запроса повторно выдаётся последний снимок `info` в шину событий, чтобы экраны (соседи, настройки) не «застывали», если свежий JSON уже есть в памяти; при **полностью недоступном транспорте** (нет BLE/Wi‑Fi) в шину снова отдаётся последний закэшированный `evt:info`.
- **Экран чата** — при возврате приложения с фона список сообщений перечитывается из локальной БД, чтобы устранить расхождение «в базе есть — в открытом чате нет»; то же при **возврате с вложенного экрана** поверх чата (например диалог), через глобальный `RouteObserver`.
- **Отправка в личный чат** — при невалидном ID получателя (не 16 hex) в лог добавляется явная причина `invalid_recipient` вместо немого `send() == false`.
- **Ping / проверка «онлайн»** — исправлена логика `sendPing`/`sendPingTracked`: раньше признак успеха почти всегда был «да», даже если BLE не принял JSON (ошибка глоталась); теперь коротко ждём ответ роутера и отличаем `request_send_failed` от ожидания pong по эфиру. После подключения `getInfo` → `getGroups` → `getRoutes` выполняются **по очереди**, чтобы реже переполнять очередь команд на узле. Авто-ping при входе в личку задержан (~520 ms). В списке чатов: таймаут ожидания pong увеличен до 20 s, сопоставление ответа с ожидаемым ID — с учётом префикса (как в экране чата).
- **Ping** — при раннем выходе из `sendPingTracked` (~650 ms) запрос в роутере всё ещё жил до ~20 s; при отсутствии `pong` завершение с `TimeoutException` шло **без слушателя** и давало необработанную ошибку в Zone (ломало ощущение «проверки связи»). Хвост `Future` глушится через `catchError`. Диалог ping в чате: сначала «Проверяем связь…», без лишнего «отправлено» перед «онлайн»; таймаут ожидания в личном чате снимает pending по тем же правилам 8/16 hex, что и `evt:pong`.
- **Traceroute / selftest** — увеличены таймауты ожидания ответа по BLE (`traceroute` 18 s, `selftest` 15 s).
- **Прошивка: `signalTest`** — увеличены задержки между отложенными `OP_PING` по соседям (реже коллизии в эфире и лавина `notify` на BLE).
- **Экран Mesh / Signal Test** — перед запуском теста запрашивается свежий `getInfo(force: true)`; при пустом списке соседей — повтор через **1 с** и подсказка «на узле нет соседей»; сопоставление `from` в `evt:pong` с соседями по **очищенному hex** (без лишних символов); окно сбора ответов **10 с**; ошибка прошивки `signal_test_no_neighbors` показывает ту же подсказку.
- **Личный чат** — убран второй авто-ping при каждом `evt:info` (оставлены вход в чат / таймер 15 s / возврат с фона); подпись «проверяем связь» без подстановки Node ID.
- **Личный чат / отправка** — если с собеседником нет pairwise-ключа (в списке соседей для него `hasKey: false`), кнопка «Отправить» скрыта; повторная попытка отправки через обходные пути показывает подсказку «ожидание ключа».
- **Диагностика** — дополнительная строка лога для цепочки прошивка → ingest → репозиторий по входящим сообщениям.
- **BLE: разбор крупного `evt:info`** — при склейке нескольких notify промежуточный `FormatException` (обрыв строки в середине JSON) больше не засоряет `[BLE_CHAIN] app_parse single_object_retry`; полный список групп по-прежнему доступен через `getGroups`.
- **Прошивка: `notifyInfo` по BLE** — в JSON ограничено число элементов в массиве `groups` (до 12), чтобы реже резать полезную нагрузку на границе MTU; пауза между chunk’ами notify увеличена с 8 до 12 ms.
- **Прошивка: `evt:pong` по BLE** — при приёме `OP_PONG` **сразу** `notifyJsonToApp`, как для `evt:delivered` после `OP_ACK` (раньше pong всегда ждал следующий `ble::update()` и из‑за этого ощущался медленнее доставки сообщений). Если мьютекс notify занят — отложенная отправка через `update()` (как раньше), один notify за тик, приоритет после `info` и **раньше тяжёлого `evt:msg`**; `cmdId` не снимается до успешной отправки (`peek` / `take` после OK).
- **Прошивка: ретраи `OP_PING` по эфиру** — меньше дублей, чем у MSG: **до трёх передач** одного запроса (первая + 2 повтора); таймаут как у ACK в очереди сообщений; таймаут считается по **max(SF модема, подсказка RSSI→SF)** — фактический TX с **фиксированным SF модема**; при приёме **`OP_PONG`** от того же узла ретраи сбрасываются (`clearPingRetryForPeer`).
- **Прошивка: HELLO / beacon** — реже лишний трафик в эфире: базовый интервал 36 с (было 30 с), при 0 соседей агрессивный режим 12 с (было 9 с), ступени после 2/5 мин без соседей 15/18 с (было 12/15 с), при одном соседе 24 с (было 20 с), при ≥6 соседей 30 с (было 24 с), жёсткий минимум между HELLO 10 с (было 8 с).
- **Приложение: `TransportResponseRouter` и `evt:pong`** — если в JSON нет `cmdId` (прошивка не вложила), ответ всё равно сопоставляется с ожидающим tracked `ping` по полю `from` и аргументу `to` команды (полный id / префикс), чтобы тикет не «висел» вечно.
- **Прошивка: JSON `evt:pong` по BLE** — буфер сериализации увеличен до 128 байт, вызов `serializeJson` с границей; при переполнении notify не отправляется (исключена обрезка JSON и «пустой» ping).
- **Прошивка / приложение: связка пинг ↔ pong** — в ответном кадре `OP_PONG` в эфире передаётся тот же `pktId`, что был в `OP_PING` (если исходный пинг с pktId); в `evt:pong` добавлено поле **`pingPktId`** (дополнительно к `cmdId` из BLE-команды). Во Flutter — `RiftLinkPongEvent.pingPktId`.

### Тесты и инфраструктура

- Тесты на ключ дедупа входящих сообщений; исправлены ожидания в тестах навигации (дублирование заголовка на тестовой странице).
- **TransportResponseRouter** — сценарий `pong` без `cmdId` для tracked-команды `ping`.

### Веб-прошивка (ESP Web Flasher)

- Обновлены манифесты и образы `*_full.bin` для Heltec V3, V3 Paper и V4.
