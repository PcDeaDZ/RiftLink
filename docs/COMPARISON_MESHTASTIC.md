# Таблица сравнения ключевых параметров: RiftLink и Meshtastic

Ориентир для аудитории, которая знает Meshtastic или «меш по LoRa» в общих чертах. **RiftLink** — отдельный стек (не форк и не совместим с эфиром Meshtastic). Сравнение не на уровне бенчмарков, а на уровне **архитектуры и продукта**.

| Параметр | RiftLink | Meshtastic | Комментарий |
|----------|----------|------------|-------------|
| **Совместимость по эфиру** | Только узлы с прошивкой RiftLink (свой формат пакетов, sync 0x5A, opcodes) | Узлы с прошивкой Meshtastic | Это разные сети; «перепрошить и поговорить с соседом на Meshtastic» **нельзя** |
| **Тип маршрутизации** | Проактивная маршрутизация **ROUTE_REQ / ROUTE_REPLY**, таблица маршрутов + broadcast с TTL | Типично широковещательный flood с ограничением хопов в их модели | У RiftLink есть явный запрос/ответ маршрута для целей unicast/mesh |
| **Роли устройств** | Один тип узла: все ноды используют общий образ под плату (Heltec / LilyGO и т.д.), без отдельной «только репитер / только клиент» прошивки как продуктовой оси | Гибкая настройка пресетов и ролей в приложении под их экосистему | У RiftLink упор на **единый протокол и приложение**, а не на зоопарк ролей на уровне SKU |
| **Служебный трафик и телеметрия** | Периодическая **TELEMETRY** (по умолчанию порядка **60 с**), плюс HELLO/beacon в логике mesh | Автоматические телеметрия и статусы — привычная картина Meshtastic | У RiftLink служебный фон задаётся прошивкой и интервалами; это **не** копия их расписания пакетов |
| **Радио и регион** | **SX1262**, регионы **EU / UK / RU / US / AU**, параметры эфира в связке с регионом и кодом | Большой выбор железа и пресетов, привычный пользователю Meshtastic UX | У RiftLink другой баланс: **свой** набор правил под свой протокол |
| **Код и экосистема** | Открытый репозиторий (см. лицензию в корне проекта), **свой** клиент (Flutter) и BLE JSON API | GPL и огромное сообщество, много устройств и интеграций | У Meshtastic шире **аппаратная** база; у RiftLink — **полный контроль** над стеком под свои релизы |
| **Ретрансляция** | Ретрансляция в рамках политики mesh (очереди, offline, TTL), без отдельного класса «только репитер» в смысле чужого продукта | Классическая mesh-модель: ретрансляция на узлах по правилам прошивки | Ключевое отличие не «да/нет», а **как именно** устроены очереди и протокол |
| **Ограничение «хопов»** | Поле **TTL** в заголовке пакета (1 байт; фактический предел зависит от стартового TTL и политики) | Часто обсуждаемый лимит порядка **7** хопов в пользовательских описаниях | Прямое сравнение чисел **не эквивалентно**: разные заголовки и семантика |
| **Приложение** | **Flutter**: Android / iOS / Web, протокол приложения ↔ устройство на **JSON по BLE** (и развитие транспорта в проекте) | Официальные и сторонние клиенты под протокол Meshtastic | Один стек «прошивка + своё приложение» без обязанности следовать чужим релизам клиента |
| **Облако / «комнаты» / MQTT** | Нет обязательного центрального сервера для ядра mesh; связь телефон ↔ узел по BLE, mesh по LoRa | Опционально MQTT и прочие интеграции в экосистеме | RiftLink задуман как **автономный mesh + свой клиент**, а не как копия их облачных сценариев |
| **Безопасность** | **X25519**, **ChaCha20-Poly1305**, группы V2, модель инвайтов и ключей под свой протокол | Своя модель ключей и каналов Meshtastic | Сравнение только на уровне **идеи** E2E; форматы и угрозы разные |
| **Прошивка и восстановление** | **PlatformIO**, веб-флешер в **`docs/flasher/`**, манифесты под поддерживаемые платы | Привычный мир Meshtastic Flasher и их инструкции | У RiftLink свой конвейер релизов и образов **`*_full.bin`** |

---

## Когда выбирать что

- **Meshtastic** — если нужна **совместимость с чужими узлами**, большое сообщество и готовые интеграции.
- **RiftLink** — если нужна **своя сеть и свой контур развития**: протокол, приложение, релизы и железо из поддерживаемого списка без подгонки под чужой эфир.

Подробности по протоколу: [PROTOCOL.md](PROTOCOL.md), по API: [API.md](API.md).
